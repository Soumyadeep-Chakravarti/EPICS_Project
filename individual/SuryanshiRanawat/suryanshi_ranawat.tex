\begin{center}
    \textbf{\Large Abstract}
\end{center}

\noindent Breast cancer affects millions of women globally, and catching it early genuinely saves lives. Our team built a machine learning system that tries to help with detection while also being transparent about how it makes decisions -- because in medicine, a black box prediction isn't helpful.

\vspace{0.5cm}
\noindent We experimented with several classification algorithms -- Logistic Regression, Decision Trees, SVM, and Random Forest -- and combined them using soft voting. The idea was that multiple models working together would be more reliable than any single one.

\vspace{0.5cm}
\noindent One thing we really focused on was making the system explainable. It doesn't just say ``malignant'' or ``benign'' -- it provides confidence scores and shows which features influenced the decision. Our best models achieved 98.25\% accuracy with ROC-AUC above 0.99.

\vspace{0.5cm}
\noindent \textbf{Keywords:} Breast Cancer Detection, Machine Learning, Ensemble Learning, Explainable AI, Data Preprocessing, Exploratory Data Analysis

\section{MY CONTRIBUTION}

\subsection{Overview}
My main job in this project was \textbf{data preprocessing and exploratory data analysis (EDA)}. Basically, I took the raw dataset that Bhavya found and got it ready for the ML models. This might sound straightforward, but good preprocessing can make or break a project -- garbage in, garbage out, as they say.

\subsection{Data Preprocessing}

Preprocessing is one of those things that doesn't sound glamorous but is absolutely critical. Here's what I focused on:

\subsubsection{Handling Missing Values}
First thing I did was check for missing data. You'd be surprised how many datasets have gaps:
\begin{itemize}
    \item Went through all 30 features systematically
    \item Checked every single row (569 of them)
    \item Good news: WDBC is squeaky clean -- not a single missing value
    \item Documented this so the team knew we didn't need imputation
\end{itemize}

\subsubsection{Data Normalization and Scaling}
This was probably the most important preprocessing step. I used standard scaling (Z-score normalization):

\begin{equation}
z_i = \frac{x_i - \mu}{\sigma}
\end{equation}

where $\mu$ is the mean and $\sigma$ is the standard deviation.

Here's what I actually did:
\begin{itemize}
    \item Applied StandardScaler from scikit-learn to all features
    \item Made sure to fit the scaler \textit{only} on training data (this prevents data leakage!)
    \item Applied the same transformation to test data
    \item Verified the scaled features had mean~0 and std~1
\end{itemize}

\subsubsection{Train-Test Split}
I set up the train-test split with stratification:
\begin{itemize}
    \item Used an 80/20 split (455 training, 114 test samples)
    \item Stratified by diagnosis to keep class ratios the same in both sets
    \item Set a random seed so results are reproducible
\end{itemize}

Stratification ensures:
\begin{equation}
\frac{|y_{train}^{(c)}|}{|\mathcal{D}_{train}|} = \frac{|y_{test}^{(c)}|}{|\mathcal{D}_{test}|} = \frac{|y^{(c)}|}{|\mathcal{D}|}
\end{equation}

\subsection{Exploratory Data Analysis}

EDA was actually the fun part -- getting to know the dataset before building models.

\subsubsection{Statistical Analysis}
I ran a bunch of statistics on all the features:
\begin{itemize}
    \item Calculated basics: mean, median, std, min, max, quartiles
    \item Looked at how spread out the values were for each feature
    \item Flagged potential outliers using IQR (some were pretty extreme)
    \item Compared statistics between benign and malignant groups -- this was really revealing
\end{itemize}

\subsubsection{Correlation Analysis}
I built correlation matrices to understand how features related to each other:
\begin{itemize}
    \item Made heatmaps for all 30 features (pretty busy-looking, honestly)
    \item Found that radius, perimeter, and area are super correlated (makes sense mathematically)
    \item Noted multicollinearity issues that might affect linear models
    \item Narrowed down to top 15 features for a cleaner visualization
\end{itemize}

The high correlations between geometric features weren't surprising -- area is basically radius squared, after all.

\subsubsection{Feature Distribution Visualization}
I created visualizations to see how benign and malignant cases differ:
\begin{itemize}
    \item Histograms comparing the two classes -- some features showed really clear separation
    \item Box plots to see spread and outliers
    \item Violin plots for more detailed distribution shapes
    \item Identified which features would likely be most predictive
\end{itemize}

\subsubsection{Key Features Identification}
From all this analysis, the most discriminating features seemed to be:
\begin{enumerate}
    \item \textbf{Worst Area}: Huge difference between benign and malignant
    \item \textbf{Worst Concave Points}: Malignant tumors have more of these
    \item \textbf{Worst Radius}: Malignant tumors tend to be bigger
    \item \textbf{Mean Concave Points}: Consistently different between classes
    \item \textbf{Worst Perimeter}: Related to tumor size
\end{enumerate}

\subsection{Visualization Development}

I made several plots that ended up in our reports:

\subsubsection{Correlation Heatmap}
The heatmap showing correlations among the top 15 features was pretty informative. You could really see clusters of related features.

\subsubsection{Feature Distribution Plots}
I created side-by-side plots comparing benign and malignant distributions. For features like ``worst concave points,'' you can literally see the two classes barely overlap -- that's a good sign for classification.

\subsubsection{Class Distribution Analysis}
I also visualized the class balance:
\begin{itemize}
    \item Benign: 357 cases (62.7\%)
    \item Malignant: 212 cases (37.3\%)
\end{itemize}

It's imbalanced, but not severely. This informed our decision to use stratification and to track metrics like recall that account for class imbalance.

\subsection{Data Quality Assurance}

Before declaring the data ``ready,'' I did some final checks:
\begin{itemize}
    \item Confirmed all columns had correct dtypes (no strings hiding in numeric columns)
    \item Searched for duplicate rows (none found)
    \item Sanity-checked feature ranges against what medical literature says is reasonable
    \item Documented every preprocessing step so others could reproduce it
\end{itemize}

%==================== PROJECT OVERVIEW ====================%

\section{PROJECT OVERVIEW}

\subsection{Problem Statement}
We wanted to build a system that could distinguish benign from malignant tumors, give confidence scores, and explain its reasoning. In medicine, just getting a ``yes'' or ``no'' isn't helpful -- doctors need to understand the why.

\subsection{System Architecture}
We organized the project into four layers:
\begin{enumerate}
    \item \textbf{Data Layer}: Loading and preprocessing (my domain)
    \item \textbf{Model Layer}: The classification algorithms
    \item \textbf{Ensemble Layer}: Soft voting combination
    \item \textbf{Output Layer}: Metrics and visualizations
\end{enumerate}

\subsection{Results Achieved}
We got some solid results:
\begin{itemize}
    \item LR and SVM both achieved 98.25\% accuracy
    \item ROC-AUC above 0.99 for the best models
    \item 98.61\% recall -- catching almost all the cancer cases
    \item Cross-validation confirmed the results were robust
\end{itemize}

I like to think my preprocessing work contributed to these numbers -- clean, well-scaled data makes everything downstream work better.

\section{CONCLUSION}

My part of this project was all about getting the data ready -- preprocessing and exploratory analysis. It doesn't sound as exciting as building ML models, but honestly, it's where a lot of the real work happens. Badly prepared data would have torpedoed everything downstream.

The EDA I did was genuinely useful. The visualizations showed which features had the clearest separation between classes, and understanding feature correlations helped avoid redundant information. The preprocessing pipeline I built followed best practices (fitting only on training data, stratified splits) to prevent the kinds of subtle bugs that can invalidate results.

I'm happy with how the project turned out -- 98.25\% accuracy is solid, and I think my careful data preparation contributed to that. It was also interesting to work on something with real medical relevance rather than just toy datasets.

This EPICS project taught me that data work is foundational. You can have the fanciest model architecture in the world, but if your data is messy or your preprocessing leaks information, none of it matters.
